# Skills Framework Design Notes

## Core Philosophy

A skill is **not** simply a bonus value attached to a hero or troop.

Instead, a skill is a collection of:

* Trigger rules
* (Activation rules - see below, migh be superflous and unnecessary)
* Conditions
* Targeting rules
* Effects
* Status applications
* Duration rules
* Stacking rules

The battle engine should never contain hero-specific or troop-specific logic.

Instead:

* The battle engine emits events.
* The skill engine reacts to events.
* Skills generate effects and statuses.
* Effects are collected and resolved into battle modifiers.

---

# Skill Lifecycle

Every skill follows the same conceptual pipeline:

```text
Trigger (i.e. when is a skill evaluated, does not mean a RNG roll/chance to trigger)
  ↓
Activation Roll (confusing - RNG could be a Condition instead)
  ↓
Condition Check
  ↓
Target Resolution
  ↓
Effect Creation
  ↓
Status Application
  ↓
Stack Resolution
  ↓
Effect Resolution
```

---

# Skills Are Effect Producers

A skill should not directly modify damage.

Instead, a skill produces one or more effects.

Examples:

```text
Known SkillMod effects:
DamageUp
DefenseUp
OppDefenseDown
OppDamageDown

THEORIZED troop skill effects:
TroopDamageUp
TroopDefenseUp

Special skill effects for handling stateful logic, modifying targeting mechanics etc:
Retarget
ApplyStatus
```

A skill may produce multiple effects simultaneously.

Example:

Saul - Taskforce Training

Produces:

* DefenseUp +15%
* HealthUp +10%

These are separate effects generated by the same skill.

---

# Effects

Effects are the smallest meaningful combat modifiers.

Examples:

```text
DamageUp
+25%

DefenseUp
+15%

OppDefenseDown
+20%
```

Effects eventually contribute to SkillMod calculations.

Effects may be:

* Immediate
* Delayed
* Conditional
* Temporary
* Permanent for battle duration

---

# Statuses

Many skills do not directly create combat modifiers.

Instead they create statuses.

A status is a persistent combat object stored in BattleState.

Examples:

```text
Cursed
Terrified
Shielded
Marked
```

Statuses:

* Have duration
* Can expire
* May refresh
* May stack

Statuses may themselves generate effects while active.

---

# Example: Petra - Evil Eye

Skill description:

```text
50% chance to curse target,
increasing damage taken by 50%.
```

This is NOT implemented as:

```text
DamageUp +50%
```

Instead:

Turn N:

* RNG succeeds, skill triggers
* Target troop type at opposing side is selected
* Cursed status is applied on that target, like queued up.

Turn N+1:

* Cursed status is active on opp. target troop type, set during previous turn.
* The status contributes effects
* Effects enter SkillMod calculation when attacks are made upon that specific opposing troop type target.

Thus:

Skill
→ Status
→ Effects
→ Damage

---

# Example: Sophia Terror

Sophia demonstrates why statuses must be first-class objects.

Terror is:

* Applied by one skill
* Referenced by another skill
* Exists independently of either skill

This means:

Statuses cannot simply be booleans.

They must be stored as combat objects with:

```python
StatusDefinition
ActiveStatus
```

and tracked in BattleState.

---

# Trigger Types

Skills are event-driven.

Examples:

```python
ALWAYS
TURN_START
PHASE_START
ATTACK
```

The trigger type determines when during a battle turn the skill is evaluated - e.g. some skills are always active from start, some are rolled per attack phase, etc.

# Conditions

Skills may require specific battlefield conditions.

Examples:

```text
Attacker must be Cavalry
Target must be Infantry
Infantry line must exist
Target must be Terrified
Turn >= 5
```

Conditions are checked after activation.

---

# Targeting

Skills should not hardcode specific enemy lines whenever possible.

Instead, they should use targeting rules.

Examples:

```text
SELF
ENEMY
SELF_INFANTRY
ENEMY_INFANTRY
CURRENT_TARGET
ATTACKER_OF_STATUS_TARGET
DEFENDER_OF_STATUS_TARGET
```

The battle engine determines actual targets.

Skills consume that information.

This is important because some mechanics (such as cavalry retargeting) can dynamically alter targets.

---

# Stacking

Effects are grouped by:

```text
Effect type
Effect operands ID
```

Same effect_op:

```text
add together
```

Example:

```text
DamageUp 101
+25%

DamageUp 101
+25%
```

becomes:

```text
DamageUp 101
+50%
```

Different effect_op:

```text
multiply
```

Example:

```text
DamageUp 101 = +50%
DamageUp 102 = +50%
```

becomes:

```text
1.5 × 1.5 = 2.25
```

This behavior is implemented through EffectCollection.

# SkillMod Philosophy

Current theory:

```text
SkillMod = (DamageUp × OppDefenseDown × TroopDamageUp × TroopOppDefenseDown) / (DefenseUp × OppDamageDown × TroopDefenseUp × TroopOppDamageDown)
```
NOTE:  
Not sure about the specific troop effect types, but Daryl has made it clear that troop skills have their own "skill ID's" separate from hero skills which MIGHT simply be same effect types (e.g. DamageUp) but different effect_op's than ANY hero skill in existicense OR separate effect types for troop skills.  
Either way, troop skills stack multiplicitavely, so should be functinoally same either way.  

The skill engine is responsible for calculating each component.

The battle engine consumes the final SkillMod.

---

# Hero Skills vs Troop Skills

Hero skills and troop skills should use the same framework.

Differences should exist only in data.

Both may have:

* Triggers
* RNG
* Conditions
* Targets
* Effects
* Statuses

Troop skills are generally simpler but should not require a separate combat architecture.

---

# Skill Engine Responsibilities

The SkillEngine should:

* Evaluate triggers
* Perform RNG rolls
* Check conditions
* Resolve targets
* Apply statuses
* Collect effects
* Resolve multipliers

The SkillEngine should NOT:

* Calculate damage
* Kill troops
* Apply casualties
* Decide battle outcome

Those responsibilities belong to the BattleEngine.

---

# Battle Engine Responsibilities

The BattleEngine should:

* Execute turns
* Execute phases
* Determine targets (i.e. targeting mechanics/algorithm)
* Request modifiers from SkillEngine
* Calculate damage
* Apply casualties
* Update BattleState
* Determine winner

The BattleEngine should treat skills as a black box.

It only asks:

```python
what effects are active right now?
```

and uses the answer in damage calculations.

---

# Long-Term Goal

HOPEFULLY a framework that allow theorycrafters to model skills by composing behavior from reusable building blocks rather than writing custom code and ad-hoc handlers for every hero (and troop) skill

Ideally, new skills become mostly data:

* Trigger
* (Activation - see above, this is prob superflous)
* Conditions
* Targets
* Effects
* Statuses

with custom handlers only required for truly unique mechanics.

This makes the simulator easier to maintain, easier to experiment with, and better suited for reverse-engineering complex battle mechanics.
